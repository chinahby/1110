1.Sensors API Message Protocol

Clients (applications) communicate with the sensors using a bi-directional messaging mechanism. Client application performs configuration and/or data request operations by sending ASCII-based messages through the IPort1 Interface (refer to Sensors SDD document 80-xxxxx for details about IPort1) interface. 

 

These messages generally have the following syntax:

·         “set, <data_type1>:<set_expr1>  [, <data_type2>:<set_expr2>  ]” ;   where

<data_type>            : one of the supported data types listed in section 2.1

<set_expr>                         : “<attribute>=<value>”; where

<attribute>              : one of the supported attribute per data type listed in section 2.1

<value>                  : the value that the attribute needs to be set using the operator

 

·         “get, <get_expr1> [,<get_expr2>…]”    where

i. <get_expr>          : “<data_type>” if the client is requesting for a data type; or

ii. <get_expr>          : “<data_type>:<attribute>” if the client requests the attribute                                                                           corresponding to the data type; or

iii. <get_expr>         : a condition statement in the format “<operand><operator><value>"; where

 

<operand>              : <data_type> if condition is set on a data type or

                                <data_type>:<attribte> if condition is set on an attribute

<data_type>            : one of the supported data types listed in section 2.1

<attribute>              : one of the supported attribute per data type listed in section 2.1

<operator>              : one of the operators listed in section 2.2

<value>                  : the operand of the condition statement

 

Note:

1. iii above represents setting a condition on the requested data. For more details about setting conditions, see section 4.

3 Request Response Structure

The client can request for data, conditional data or set value for a certain attribute on the sensors. The client creates the request message based on the syntax defined in section 1 and passes it as message buffer to IPort1_Write API (see Sensors SDD for more details). The client can then use the IPort1_Read API to retrieve the response message from the buffer returned by the Sensors Manager. Currently we support usage of 2 commands in the message string: get and set.

 

The Sensors Manager upon receiving the request message will validate the request and send a response to the client. The response string will echo the request so the client can distinguish the requested command if it is sending multiple requests. It will also have an “error” field that indicates whether there is some error in the requested data. More on error codes in section 3.1.

If the client sent a “get” command, the error field will be followed by the actual data. If the client sent a “set” command, the response will echo the request string and attach the error field indicating if the set was successful or not. The echoed request, error and the response are delimited by # (pound) character.

 

If the request was a get, the response will be

“error = xxxx # response# get, request ”; where

xxxx                 : one of the error codes listed in section 3.1

 

Example:

Request:

“get, datatype_A, datatype_B, datatype_A:AttributeA”

Response:

“error = xxxx #

datatype_A = value_A, datatype_B = value_B, datatype_C:AttributeA = value_C # get, datatype_A, datatype_B, datatype_A:AttributeA ”

 

           

For cases where there may be more than one value in the response for a given request, the individual fields in the response string will be separated by a semicolon (;). The ordering of the response fields should be maintained as listed in the table in section 2.1.One example of such response structure will be for gpsinfo data type.

 

Example:

Request:

“get, datatype_A”

Response:

“error = xxxx #

datatype_A = value_A; value_B; value_C … # get, datatype_A

 

 

 

If the request was a set, the response will be

“set, error = xxxx #request”

xxxx                 : one of the error codes listed in section 3.1

 

Example:

Request:

“set, datatype_A: AttributeA = Value1 , datatype_B: Attribute_B = Value2”

Response:

“error = xxxx # set, datatype_A: AttributeA = Value1 , datatype_B: Attribute_B = Value2”

3.2         Auto Generation of Data

 

The “set” command if successful will not result in any data being generated from the sensor. To generate data, the client needs to issue the “get” command and request the required data. The “get” command by default will generate just one instance of data. To auto generate multiple instances, the client will have to set the frequency attribute for the data type for which it requires auto generation.

 

Example:

 

Single Shot Data Generation

The client is requesting for datatypeA. Since the client is not setting the frequency, only one sample is generated.

Request:

“get, datatype_A”

Response:

“error = xxxx # datatype_A = value_A # get, datatype_A”

 

Auto Generation

Here the client is setting the frequency at which it wants datatype_A to be generated (20 Hz). If the frequency could be set successfully; the client can request for datatype_A and expect a sample to be generated every 20Hz.

Request:

“set,datatype_A:freq=20”

Response:

“error=0#set,datatype_A:freq=20” // 0 = SUCCESS

Request:

“get, datatype_A”

Response:

“error=xxxx#datatype_A=value_A#get,datatype_A”

“error=xxxx#datatype_A=value_A#get,datatype_A”

“error=xxxx#datatype_A=value_A#get,datatype_A”

…

To stop auto-generation of data, the client will have to set the frequency at which it wants datatype_A to 0.

Request:

“set,datatype_A:freq=0”

Response:

“error=0#set,datatype_A:freq=0”

 

 

3.3         Calibration

 

 

The “set” command can also be used to calibrate the sensor. Only certain sensors can be calibrated. See section 2.1 for data types that have the calibrate attribute supported. Setting calibrate =1 starts calibration. Calibrate = 0 stops calibration.

Example:

Start Calibration

            Request:

                        "set,datatype_A:calibrate=1"

            Response:

                        "error=0#set,datatype_A:calibrate=1"

 

Stop Calibration

            Request:

                        "set,datatype_A:calibrate=0"

            Response:

                        "error=0#set,datatype_A:calibrate=0"

 

4 Conditional Data Generation

Clients can request for data based on certain conditions. The operators that are used in section 2.2 are used specifically to set conditions on the requested data. The clients get a response from the Sensors Manager only once the condition is met. The client should use the “get” command to set conditions. Conditions can be set both on data types as well as attributes. See section 1 for the prototype for requesting conditional data.

 

The clients can set multiple conditions in the same command or set conditions on different commands. The clients can also set multiple conditions on the same data type in the same command. The general rule of thumb is that if multiple conditions are set using the same command, the conditions are ANDed irrespective of whether the conditions are set on the same data type or different data types. If the client sets multiple conditions using different commands, the conditions are ORed.

 

The client can also cancel conditions that are set. The cancellations are done on the data types on which the conditions are set. In other words if datatype_A has 2 different conditions set, cancelling the condition on datatype_A will cancel both instances of the set conditions. Also if multiple conditions are set using the same command, cancelling one of the data types in that command will cancel all the conditions in that command. The keyword “off” shall be used to cancel conditions. If the client wishes to cancel all the conditions it has set, it can use the keyword “all”.

 

The client after setting a condition on a data type can also request the same data type (for either single shot or auto generated data) in a separate get command. The Sensors Manager differentiates the two streams of data.

 

We do not support setting conditions on system attributes (ts, sysres etc)

 

 

Example:

1. Setting a condition

            Request:

                        "get,datatype_A>10"

            Response:

                        "error=0#datatype_A=11#get,datatype_A>10"

 

2. Setting multiple conditions on different data types using same command.  Note that conditional data is generated only when all conditions are satisfied.

            Request:

                        "get,datatype_A>10,datatype_B<5,datatype_C>=4"

            Response:

                        "error=xxxx #

                         datatype_A=12,datatype_B=4,datatype_C=4#

get,datatype_A>10,datatype_B<5,datatype_C>=4"

 

3. Setting multiple conditions on same data type using same command. Again data is generated only when all conditions are satisfied

            Request:

                        "get,datatype_A>10, datatype_A!=12"

            Response:

                        “error=xxxx #

                         datatype_A=11#

get, datatype_A>10, datatype_A!=12"

 

 

3. Setting multiple conditions on same data types using different command. Note that conditional data is generated if either condition is satisfied.

            Request:

                        "get, datatype_A>10”

                        “get, datatype_A=15”

            Response:

                        "error=xxxx#datatype_A=12#get,datatype_A>10“

 

4. Cancelling a condition on datatype_A set in example 2

            Request:

                        "get,datatype_A=off"

            Response:

                        “error =xxxx# get,datatype_A=off"

 

 

5. Cancelling all the set conditions in example 2

            Request:

                        "get,all=off"

            Response:

                        “error=xxxx# get,all=off"

      4. Accelerometer Use Cases

Operation

Request

Response

Set Frequency on accelx

"set accelx:freq=20"

"error=0#set,accelx:freq=20"

Set frequency on accely and accelz

"set,accely:freq=20, accelz:freq = 50"

"get ,accelx>10, accely>20"

Get Data

Since we are requesting accelx, accely and accelz at different frequencies, the samples will be generated based on the frequency set on the data types. But each sample will not contain all the data requested. If no data was available at the time the same was returned, the data type will have the value na. Also each sample will have the error “DATA_NOT_AVAILABLE” since some of the requested data is not available.

"get,ts,accelx,accely, accelz"

At time = 20 ms

"error=12 #

ts=20, accelx=na, accely=na, accelz=3#

get,ts,accelx, accely,accelz "

At time = 40 ms

"error=12 #

ts=40, accelx=na, accely=na, accelz=4#

get,ts,accelx, accely,accelz "

At time = 50 ms

"error=12 #

ts=50, accelx=4, accely=12, accelz=na#

get,ts,accelx, accely,accelz "

At time = 50 ms

"error=0 #

ts=100, accelx=4, accely=12, accelz=15 # get,ts,accelx, accely,accelz "

Set Frequency for accelx to 0

Since we set frequency of accelx to 0, and we are requesting accelx accely and accelz in the same command above, all future samples will have accelx set to na and will be returned with an error of DATA_NOT_AVAILABLE

"set,accelx:freq=0"

"error=0#set,accelx:freq=0"

Stop Condition

Had the client cancelled the condition only on accelx, it would still have cancelled both the conditions since it translates to an AND

"get,accelx=off, accely=off"

"error=0#get,accelx=off,accely= off"

Stop Condition (all)

"get, all = off"

"error=0# get, all = off"
